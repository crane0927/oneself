# 原则 6 检查清单（RESTful API 设计）

**依据**：宪法原则 6（.specify/memory/constitution.md 第 115–133 行）及 data-model.md 规则集。

审计时对每个接口逐项勾选，判定：**通过** / **不通过**（附原因与建议）/ **例外**（附理由）。

---

## C1：HTTP 方法标准

- **通过条件**：方法属于 GET、POST、PUT、DELETE、PATCH
- **不通过时建议**：改用标准方法（如查询用 GET）
- 判定：□ 通过　□ 不通过　□ 例外

---

## C2：URL 名词化、无动词

- **通过条件**：路径为名词（复数）且不含 get/create/delete/list 等动词
- **不通过时建议**：改为资源路径（如 /users/{id}）+ 合适方法
- 判定：□ 通过　□ 不通过　□ 例外

---

## C3：响应 JSON + Result 包装

- **通过条件**：响应体为 JSON 且使用统一 Result&lt;T&gt; 包装
- **不通过时建议**：统一使用项目 Result 类型
- 判定：□ 通过　□ 不通过　□ 例外

---

## C4：分页与排序

- **通过条件**：分页用 page、size，排序用 sort（或文档化例外）
- **不通过时建议**：改为 GET + page/size/sort 或记录例外
- 判定：□ 通过　□ 不通过　□ 例外

---

## C5：版本控制

- **通过条件**：URL 含版本（如 /api/v1/）或文档说明策略
- **不通过时建议**：采用路径版本或文档说明
- 判定：□ 通过　□ 不通过　□ 例外

---

## C6：文档标注

- **通过条件**：API 文档中标注该接口的 HTTP 方法与主要状态码
- **不通过时建议**：更新 API 文档
- 判定：□ 通过　□ 不通过　□ 例外
